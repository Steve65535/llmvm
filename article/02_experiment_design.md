# 实验设计方案

## 1. 实验目标

验证 LLMVM 在以下方面的性能：
1. **算法实现能力**：能否正确实现经典算法
2. **复杂推理能力**：能否解决需要多步推理的问题
3. **可扩展性**：性能如何随任务复杂度变化
4. **注意力机制的有效性**：两阶段选择是否优于基线
5. **实际应用价值**：能否完成真实世界的任务

---

## 2. Benchmark 任务设计

### 2.1 算法实现 (AlgoEval)

**目标**：测试 LLMVM 实现经典算法的能力

**任务列表**：

| 类别 | 任务 | 难度 | 评估指标 |
|------|------|------|---------|
| 排序 | 快速排序 | 中 | 正确性、代码质量 |
| 排序 | 归并排序 | 中 | 正确性、代码质量 |
| 搜索 | 二分搜索 | 易 | 正确性、边界处理 |
| 搜索 | 深度优先搜索 | 中 | 正确性、完整性 |
| 图算法 | Dijkstra 最短路径 | 难 | 正确性、效率 |
| 图算法 | 拓扑排序 | 中 | 正确性、边界情况 |
| 动态规划 | 背包问题 | 难 | 正确性、优化程度 |
| 动态规划 | 最长公共子序列 | 中 | 正确性、空间优化 |
| 字符串 | KMP 字符串匹配 | 难 | 正确性、理解深度 |
| 数据结构 | 实现平衡二叉树 | 难 | 正确性、完整性 |

**评估标准**：
```python
def evaluate_algorithm(implementation, test_cases):
    scores = {
        'correctness': 0,      # 通过测试用例的比例 (0-100)
        'code_quality': 0,     # 代码可读性、注释 (0-10)
        'efficiency': 0,       # 时间复杂度是否最优 (0-10)
        'robustness': 0,       # 边界情况处理 (0-10)
    }
    
    # 运行测试用例
    passed = sum(run_test(implementation, tc) for tc in test_cases)
    scores['correctness'] = (passed / len(test_cases)) * 100
    
    # 代码质量评估（人工或 GPT-4 评分）
    scores['code_quality'] = evaluate_code_quality(implementation)
    
    # 效率评估
    scores['efficiency'] = evaluate_complexity(implementation)
    
    # 鲁棒性评估
    scores['robustness'] = evaluate_edge_cases(implementation)
    
    return scores
```

### 2.2 复杂推理 (ReasonEval)

**目标**：测试多步推理和问题分解能力

**任务来源**：
- **HumanEval**：编程问题（164 个任务）
- **MATH**：数学问题（12,500 个问题）
- **GSM8K**：小学数学应用题（8,500 个问题）
- **MMLU**：多领域选择题（57 个子任务）

**评估指标**：
- **准确率**：正确答案的比例
- **推理步骤数**：平均需要多少步
- **Token 消耗**：平均每个任务的 Token 数
- **执行时间**：平均完成时间

### 2.3 实际应用 (RealWorldEval)

**目标**：测试在真实场景中的表现

**任务设计**：

#### 任务 1：数据分析流水线
```
输入：CSV 数据文件 + 分析需求
输出：清洗后的数据 + 可视化图表 + 分析报告

子任务：
1. 读取并理解数据结构
2. 数据清洗（缺失值、异常值）
3. 探索性数据分析
4. 生成可视化图表
5. 撰写分析报告
```

#### 任务 2：Web 应用开发
```
输入：功能需求描述
输出：完整的 Web 应用（前端 + 后端）

子任务：
1. 需求分析和架构设计
2. 数据库设计
3. 后端 API 实现
4. 前端界面实现
5. 测试和调试
```

#### 任务 3：系统运维任务
```
输入：服务器日志 + 问题描述
输出：问题诊断报告 + 修复脚本

子任务：
1. 日志分析
2. 问题定位
3. 编写修复脚本
4. 验证修复效果
5. 生成报告
```

**评估标准**：
- **任务完成度**：0-100%
- **代码质量**：可读性、可维护性（1-10）
- **用户满意度**：人工评估（1-10）
- **效率**：完成时间、LLM 调用次数

## 3. 基线方法

### 3.1 对比系统

| 系统 | 类型 | 实现方式 |
|------|------|---------|
| **GPT-4 (直接)** | 单轮 | 直接提问，一次性回答 |
| **GPT-4 + ReAct** | 多轮 | Thought-Action-Observation 循环 |
| **Claude 3.5 Sonnet** | 单轮 | 直接提问 |
| **AutoGPT** | 自主代理 | 目标驱动的自主执行 |
| **LangChain** | 框架 | 使用 Agent + Tools |
| **Tree of Thoughts** | 搜索 | 广度优先搜索思维树 |
| **LLMVM (ours)** | 图灵完备 | 任务树 + 注意力机制 |

### 3.2 公平性保证

所有系统使用：
- **相同的 LLM 后端**：GPT-4 或 Claude 3.5
- **相同的工具集**：文件操作、命令执行、网络请求
- **相同的超参数**：温度、top_p 等
- **相同的评估数据**：同一批测试用例

## 4. 消融实验 (Ablation Study)

**目标**：验证每个组件的必要性

### 4.1 实验变体

| 变体 | 说明 | 预期影响 |
|------|------|---------|
| **LLMVM-Full** | 完整系统 | 基准 |
| **LLMVM-NoAttention** | 去掉注意力机制 | 上下文溢出，性能下降 |
| **LLMVM-NoHistory** | 去掉命令历史 | 无法利用过去执行结果 |
| **LLMVM-NoScope** | 去掉变量作用域 | 变量冲突，错误增加 |
| **LLMVM-FlatTree** | 限制树深度为 1 | 无法分解复杂任务 |
| **LLMVM-NoLoop** | 禁用 Loop 节点 | 无法处理迭代任务 |

### 4.2 评估指标

对每个变体测试：
- **AlgoEval**：算法实现准确率
- **ReasonEval**：推理任务准确率
- **上下文长度**：平均 Prompt 长度
- **成本**：平均 Token 消耗

## 5. 可扩展性实验

### 5.1 任务复杂度梯度

设计不同复杂度的任务：

| 级别 | 树深度 | 节点数 | 示例任务 |
|------|--------|--------|---------|
| L1 | 1-2 | 3-5 | 简单计算 |
| L2 | 2-3 | 6-10 | 单文件编程 |
| L3 | 3-4 | 11-20 | 多文件项目 |
| L4 | 4-5 | 21-40 | 小型应用 |
| L5 | 5+ | 41+ | 复杂系统 |

### 5.2 测量指标

对每个级别测量：
- **成功率**：任务完成的比例
- **平均时间**：完成时间
- **LLM 调用次数**：总调用次数
- **Token 消耗**：总 Token 数
- **内存使用**：峰值上下文长度

### 5.3 预期结果

绘制曲线：
```
成功率 vs 任务复杂度
时间 vs 任务复杂度
Token 消耗 vs 任务复杂度
```

**假设**：
- LLMVM 在高复杂度任务上优势更明显
- 注意力机制使 Token 消耗保持线性增长

## 6. 注意力机制专项实验

### 6.1 实验设计

**变量**：
- 任务树节点数：10, 20, 50, 100, 200
- 注意力选择策略：
  - 无选择（全量上下文）
  - 最近 k 个节点
  - 随机 k 个节点
  - **两阶段注意力选择（ours）**

**固定参数**：
- k = 10（选中节点数）
- 相同的任务集

### 6.2 评估指标

| 指标 | 说明 |
|------|------|
| **准确率** | 任务完成的正确性 |
| **上下文长度** | 平均 Prompt Token 数 |
| **选择准确性** | 选中节点的相关性（人工标注） |
| **成本** | 总 Token 消耗 |

### 6.3 预期结果

- **两阶段注意力** 在准确率上接近全量上下文
- 但上下文长度显著降低（50-70% 减少）
- 优于简单的启发式方法（最近 k 个、随机）

## 10. 预期挑战与应对

### 10.1 LLM 的不确定性
- **问题**：相同输入可能产生不同输出
- **应对**：每个任务运行 3-5 次，取平均值

### 10.2 评估的主观性
- **问题**：代码质量、用户满意度难以量化
- **应对**：多位评审员独立评分，计算一致性

### 10.3 成本控制
- **问题**：大规模实验的 API 成本高
- **应对**：
  - 使用更便宜的模型做初步筛选
  - 缓存重复的 LLM 调用
  - 优先测试关键场景
